Java — the ideas your codebase is built on

Examples mirror real platform patterns with names simplified for study. Answer the hidden question in your head before you tap.

Foundations

References: two variables, one object

Primitives (int, boolean…): the variable is the value. Everything else: the variable points to an object on the heap. Two references can share one object — mutate through one, the other sees it.

List<String> x = new ArrayList<>();
List<String> y = x;      // same object, two names
y.add("hello");
x.size();                 // 1 — x sees it

Yes. The method got a copy of the reference, not the list — both point at the same heap object. This is why helpers can mutate DTOs you pass in, and why defensive copies exist.

Foundations

Stack, heap, and garbage collection

Each thread gets a stack: one frame per method call holding locals and parameters — created on call, destroyed on return. Objects live on the shared heap and survive as long as anything references them. The garbage collector frees heap objects nothing points to anymore.

  • A method's local variables can never be seen by another thread — that's why locals are automatically thread-safe.
  • A "memory leak" in Java = objects still referenced that you'll never use again (a static Map that only ever grows, an unbounded cache).
  • Stack trace = a snapshot of one thread's stack, frame by frame. You already read these daily.

The field lives on the heap inside one shared singleton object — both request threads see it. Locals live in each thread's own stack frame; there's physically no sharing.

Foundations

== vs .equals() — and the hashCode contract

a == b            // same object?  (identity)
a.equals(b)       // same content? (what you almost always mean)
"ACTIVE".equals(status)   // null-safe: literal first

The contract: if two objects are equal, they must return the same hashCode(). HashMap/HashSet find things by hash bucket first, equals second — break the contract and your object goes in but never comes back out.

Default identity semantics apply: two "identical" keys are different objects with different hashes, so get misses what put stored. Lookups mysteriously return null. Use records, or override both, or don't key on it.

Design

Immutability + records

An immutable object can't change after construction — no setters, final fields. It's automatically thread-safe, safe to share, safe to cache, safe to use as a map key. Java records give you it in one line:

public record CredentialDto(String ref, String type, String status) {}
// constructor, getters, equals, hashCode, toString — all generated, all correct

Rule of thumb: DTOs and value objects → immutable. Entities with a lifecycle (a credential moving through states) → mutable, but only through guarded methods.

Race conditions are two threads seeing a value mid-change. No change, no "mid." There is nothing to synchronize because there is nothing to protect.

Design

Enums as state machines

public enum CredStatus {
    CREATED, PENDING_ACTIVATION, ACTIVE, SUSPENDED, REVOKED;

    public boolean canSign() { return this == ACTIVE; }

    public boolean canBecome(CredStatus next) {
        switch (this) {
            case CREATED:            return next == PENDING_ACTIVATION;
            case PENDING_ACTIVATION: return next == ACTIVE || next == REVOKED;
            case ACTIVE:             return next == SUSPENDED || next == REVOKED;
            case SUSPENDED:          return next == ACTIVE || next == REVOKED;
            default:                 return false;
        }
    }
}

PENDING_ACTIVATION — created, awaiting proof-of-possession. canSign() only passes for ACTIVE. A failure mid-verification parks it here forever: a real production bug class.

Design

Interfaces + the Factory pattern

An interface is a contract; a factory picks which implementation honors it. Callers never write if (type == …) — each type's rules live in its own class.

interface CredHelper { void validateForAdd(String ref); void bind(String ref); }

class HardwareTokenHelper implements CredHelper { /* serial checks */ }
class PhoneHelper         implements CredHelper { /* SMS checks */ }

static CredHelper forType(String type) {
    switch (type) {
        case "hw_tok": return new HardwareTokenHelper();
        case "oob":    return new PhoneHelper();
        default: throw new IllegalArgumentException(type);
    }
}

One new class + one factory case. Zero changes to callers or existing helpers. That isolation is the point — the Open/Closed Principle in practice, and the architecture of your credential and proofing subsystems.

Chunking payoff: once you see "one contract, one dispatcher, one class per type," a 30-file package reads as three things, not thirty.

Generics: read one layer at a time

List<Credential> = a list that only holds Credentials, checked at compile time. When a signature scares you, cover the angle brackets with your hand and uncover one layer at a time:

ResponseEntity<ApiResponse<List<CredDto>>> r;
// "a ResponseEntity…            (what the HTTP layer returns)
//    containing an ApiResponse…  (the envelope)
//      containing a list of CredDto" (the payload)

    return list.isEmpty() ? null : list.get(0);
}

A map from String keys… to lists… of maps from String to Object. (Layer by layer — never all at once. And if you ever design one of these yourself, make a named class instead.)

Streams: loops as pipelines

Filter chooses (fewer elements, same type); map converts (same count, possibly new type).

Tools

Optional: absence with a seatbelt

Robustness

Exceptions: never swallow, always wrap

try {
    helper.bind(ref);
} catch (BindingException e) {
    throw new ApiException("binding failed for ref=" + ref, e);  // context + cause
} finally {
    releaseConnection();   // runs no matter what
}

Checked = expected failures the caller must handle. Unchecked = bugs. Reading a trace: jump to the last Caused by:, then the first line in your own package.

It drops the stack trace and cause chain — "null" is all you'll know. Log the exception object, or wrap and rethrow. Swallowed exceptions turn one-hour investigations into one-week ones.

Robustness

Thread safety in one rule

Every HTTP request runs on its own thread; Spring services are singletons shared by all of them. So: no per-request state in service fields. Per-request data lives in parameters and locals (which are per-thread by construction — see the stack/heap card).

Two simultaneous requests overwrite each other — request A acts on B's user. Intermittent, invisible in dev, catastrophic in prod. The classic shared-singleton race.

Spring Boot — the container, demystified

One mental model explains 90% of Spring: at startup, a container scans for your classes, creates one instance of each, and wires them into each other's constructors.

Core

What the container does at startup

  1. Scan packages for @Component / @Service / @Repository / @RestController
  2. Create one instance of each (a "bean" — singleton by default)
  3. Inject: constructors that need other beans get them passed in
  4. Auto-configure from the classpath: web starter present → embedded Tomcat + JSON, ready
@Service
public class CredService {
    private final CredRepository repo;         // final + constructor = injected
    public CredService(CredRepository repo) { this.repo = repo; }
}

The class declares what it needs, not how to build it. Tests hand it a fake; swapping implementations touches no callers. It's the factory pattern applied to the whole app, automatically.

Structure

Three layers, strictly separated

controller/   HTTP in/out ONLY — thin
service/      business rules, orchestration, transactions
repository/   database access ONLY
dto/          the API's shape      entity/  the table's shape

Reading order for any endpoint: controller (30 seconds), the one service method (the logic), the repository calls (what persists). Never start in a util class.

API contract and DB schema change for different reasons. Separation means a new column doesn't leak into the API and an API rename doesn't demand a migration — the service mapping absorbs change.

Structure

@Transactional: all or nothing

@Transactional
public void bindCredential(String userRef, String credRef) {
    credRepo.setStatus(credRef, "ACTIVE");
    eventRepo.insert(new AppEvent("credAddOk", userRef));
    // both commit together — an exception rolls back BOTH

No transaction. Spring wraps beans in a proxy; only external calls pass through it. this.method() bypasses the proxy. Classic interview question, classic real bug.

Triage smell: "state changed but no event row" (or the reverse) = writes that should have shared a transaction and didn't.

Operations

Where configuration comes from — three layers

  • Properties / config server — deployment config: ports, URLs, per-environment values
  • Key Vault / secret store — credentials and connection strings, resolved at startup
  • Config database — per-client business flags, changeable at runtime without redeploy

Config database — per-client, business-level, no redeploy. Properties are per-environment (not per-client); the vault is for secrets, not behavior. Knowing WHICH layer holds a setting is half of config triage.

Operations

Bean scope + lifecycle (the parts that matter)

  • Singleton (default): one instance forever → must be stateless per-request.
  • Startup order: construct → inject → @PostConstruct init hooks → ready. Slow startup? Something's init hook is doing real work (loading config, warming caches).
  • @Value("${my.prop}") and @ConfigurationProperties pull property values into beans at construction time — config is frozen into the bean at startup unless the app explicitly supports refresh.

The value was injected at startup and lives in the bean's field now. Without a refresh mechanism (or a runtime config DB read), property changes need a restart. "Did it reload?" is a real triage question.

request ──► filter: extract "Authorization: Bearer <jwt>"

SQL — set thinking, not loops

A query describes a result; the engine finds the how. Tables here are simplified stand-ins: USERS, CREDS, EVENTS, CLIENTS.

Model

The real execution order

you write:        it actually runs:
SELECT   (5)      1. FROM + JOINs   build the row set
FROM     (1)      2. WHERE          filter rows
WHERE    (2)      3. GROUP BY       collapse into groups
GROUP BY (3)      4. HAVING         filter groups
HAVING   (4)      5. SELECT         pick/compute columns
ORDER BY (6)      6. ORDER BY, TOP  sort, cut

WHERE runs at step 2; aliases are born at step 5. Same logic: aggregate filters need HAVING because WHERE runs before groups exist.

Joins

Inner filters, left preserves

-- ALL active users, flagging who has NO credential:
SELECT u.REF
FROM USERS u
LEFT JOIN CREDS c ON c.USER_ID = u.ID
WHERE u.STATUS = 'ACTIVE'
  AND c.ID IS NULL;        -- the "no match" detector
  • WHERE on the right table of a LEFT JOIN silently turns it INNER — right-table conditions go in ON.
  • Fan-out: 1 user × 3 creds = 3 rows. Inflated counts? Count the base table alone first.

LEFT JOIN to creds (phone-type condition inside ON), then WHERE c.ID IS NULL. Absence is invisible to INNER joins; LEFT + IS NULL is how you query for what's missing.

Traps

NULL is a virus

  • col = NULL is never true → IS NULL
  • WHERE status <> 'REVOKED' also drops NULL-status rows
  • NOT IN (subquery) returns nothing if the subquery has one NULL → use NOT EXISTS
  • COUNT(*) counts rows; COUNT(col) counts non-nulls — different answers

NULLs — comparisons with NULL are unknown, and WHERE drops unknowns. Check what WHERE col IS NULL returns.

Aggregation

Group, then the conditional-count ace

SELECT cl.NAME,
       COUNT(*)                                              AS total,
       SUM(CASE WHEN u.STATUS = 'ACTIVE' THEN 1 ELSE 0 END) AS active,
       SUM(CASE WHEN u.STATUS = 'LOCKED' THEN 1 ELSE 0 END) AS locked
FROM USERS u JOIN CLIENTS cl ON cl.ID = u.CLIENT_ID
GROUP BY cl.NAME
HAVING COUNT(*) > 100;

Multiple counts, one pass. WHERE filters rows before grouping (cheap); HAVING filters groups after.

Power move

ROW_NUMBER: "latest X per Y"

WITH ranked AS (
    SELECT p.*, ROW_NUMBER() OVER (PARTITION BY p.USER_ID
                                   ORDER BY p.ID DESC) AS rn
    FROM PROOFING_XN p
)
SELECT * FROM ranked WHERE rn = 1;

Window functions compute across rows without collapsing them. If you learn one advanced SQL move, make it this — it answers every "most recent status per user" triage question.

GROUP BY collapses the row — you get the max id but lose that row's other columns without a second join back. ROW_NUMBER keeps whole rows and labels their rank.

Craft

Build queries incrementally (CTEs)

WITH recent_fails AS (
    SELECT * FROM EVENTS
    WHERE TYPE = 'authnFail'
      AND CREATE_TIME >= DATEADD(HOUR, -24, GETDATE())
)
SELECT u.REF, COUNT(*) AS fails
FROM recent_fails f JOIN USERS u ON u.REF = f.ACTOR
GROUP BY u.REF ORDER BY fails DESC;

Run the inner piece, eyeball rows, wrap the next layer, run again. Never write 40 lines and then debug. Name steps with CTEs so the query reads top-down like prose.

Performance

Index instincts

  • An index is a sorted lookup: WHERE indexed_col = x → seek (fast). A function around the column (LTRIM(col) = x) or leading wildcard (LIKE '%x') → scan (slow).
  • Giant audit/event tables: always time-range + TOP. An unbounded scan on a hot table is itself an incident.
  • Unsure of size? COUNT(*) before SELECT *.

Full scan = O(n) over hundreds of millions of rows while holding I/O and locks. The indexed time-bounded version is O(log n + k). Big-O is why one query is an incident and the other is instant.

Correctness

Transactions & isolation (enough to reason)

  • A transaction makes several statements atomic: all commit or none do — the DB-side twin of @Transactional.
  • While a write transaction holds locks, other writers to the same rows wait. Long transactions + hot rows = timeouts.
  • Two services writing the same row aren't coordinated by Java — the database serializes them. That's the platform's real concurrency control.

The database — an atomic UPDATE … SET cnt = cnt + 1 serializes on the row lock. Read-then-write in Java (read 3, write 4) can still lose updates; push the arithmetic into the UPDATE.

DSA — patterns, not memorized solutions

The protocol

7 steps, written down, every time

  2. Tiny example by hand — 3–4 items, solve it with your eyes

Skeleton

HashMap one-pass (Two Sum shape)

Map<Integer,Integer> seen = new HashMap<>();  // value → index
for (int i = 0; i < nums.length; i++) {
    int need = target - nums[i];
    if (seen.containsKey(need)) return new int[]{seen.get(need), i};
    seen.put(nums[i], i);   // check BEFORE store — why? [3,3] target 6
}
Skeleton

Sliding window: grow, shrink, record

int left = 0, best = 0;
for (int right = 0; right < n; right++) {
    // GROW: bring s[right] into window state
    while (/* window broke its rule */) {
        // SHRINK: remove s[left] from state
        left++;
    }
    best = Math.max(best, right - left + 1);  // RECORD
}

"N failures within M minutes locks the account" — a windowed count over an event stream. Fail counters and lock-until timestamps are its stored form.

Skeleton

Binary search: halve the space

int lo = 0, hi = arr.length - 1;
while (lo <= hi) {
    int mid = lo + (hi - lo) / 2;      // overflow-safe
    if (arr[mid] == target) return mid;
    if (arr[mid] < target) lo = mid + 1; else hi = mid - 1;
}
return -1;

Works on anything monotonic, not just arrays: "smallest rate that finishes in time" — if rate X works, every rate above X works → binary search the answer.

Skeleton

Tree recursion: the leap of faith

int maxDepth(TreeNode n) {
    if (n == null) return 0;                 // base case FIRST
    return 1 + Math.max(maxDepth(n.left),     // TRUST the call
                        maxDepth(n.right));   // combine
}

Simulating every nested call in their head — working memory overflows. Antidote: write the base case, then the recursive case as if the call already works. Verify on a 3-node tree only; induction does the rest.

Architecture — how ~20 services make one platform

The senior difference isn't knowing more syntax — it's holding the system's shape in your head: what talks to what, sync or async, and where truth lives.

Shape

Why microservices at all

One service per business capability (auth, credentials, MFA, orgs, roles, email…), each independently deployable with its own release cadence. The costs are real too: network calls can fail, debugging spans log files, and consistency needs design (see saga). A monolith isn't "wrong" — it's a different trade. Your platform runs both: legacy monoliths and modern services, sharing databases.

(1) Debugging: one user action leaves traces in several services' logs — you correlate by IDs and timestamps. (2) Transactions: no single DB commit spans services — you need sagas/events and idempotency. In exchange: independent deploys, isolated failures, per-team ownership.

Shape

Every service, same skeleton

gateway/LB ─► [filter: authn] ─► controller ─► service ─► repository ─► DB
                                        │
                                        └─► REST client ─► another service
                                        └─► publish event ─► message bus

Learn ONE service deeply and you've learned the reading order for all of them: find the controller, follow one endpoint down, note which tables and which outbound calls. A new service is a 1-hour read once you own the skeleton.

Communication

Sync vs async — the fundamental fork

Synchronous (HTTP)Asynchronous (message bus)

Three hops, in order: (1) did the producer write/publish the event? (event table / producer log) (2) did the consumer receive it? (handler log) (3) did the email service render and send? (its log / provider response). The failure is always at exactly one hop.

Communication

Events, at-least-once delivery, idempotency

Message buses promise at-least-once delivery — meaning: sometimes twice. Consumers must be idempotent: processing the same message twice must leave the same result as once.

  • Techniques: check-before-insert, unique constraints on a message/event ID, "set status = X" (naturally idempotent) instead of "increment" (not).
  • Every state change also writes an audit event row — that trail is your timeline in every investigation.

A redelivered message inserts a duplicate row (double email, double credential). Upsert/unique-key makes the second delivery a no-op. Design every consumer assuming duplicates WILL arrive.

Consistency

The saga pattern: transactions across services

No single DB transaction can span services. A saga breaks one business operation ("provision this user") into steps, records progress in a saga table, and defines what to do when step 3 of 5 fails: retry it, or run compensations that undo steps 1–2.

Exactly which step failed and that later steps never ran — the saga table is a checkpoint log. The fix is usually: resolve the failing step's cause, then resume or compensate. Never hand-edit the end state and leave the saga inconsistent.

Consistency

State machines everywhere

credential:  CREATED ─► PENDING_ACTIVATION ─► ACTIVE ─► SUSPENDED ⇄ ACTIVE
                              │                              │
                              ▼ (stuck = bug class)          ▼
                           REVOKED ◄─────────────────────────┘

proofing:    START ─► TERMS ─► CHECKS ─► [type-specific steps] ─► ACTIVATE ─► DONE

Every arrow = a guarded method + an audit event + often an async message. Most "weird state" tickets are an arrow that half-completed: the status column and the audit trail disagree about which arrow fired.

The transition wasn't atomic: event written, status write failed (or vice versa) — two writes that needed one transaction. Find the code path where they're separated; that's the bug.

Evolution

Legacy + modern, side by side (the strangler pattern)

You don't rewrite a 20-year platform; you grow a new system around it. The pattern: new services wrap the old engine's libraries, both write the same database, and a per-record version flag decides which code path owns each record.

  • First triage question for any record: which version owns it? The flag tells you which codebase to read.
  • Bugs love the seam: flows where a record starts in the old system and finishes in the new one.
  • Shared DB means the schema serves two masters — changes must not break either.

Two code paths implement the feature; the users' records carry different version flags, so each ran a different implementation. Confirm the flag first, then read the RIGHT codebase — hours saved.

Evolution

Multi-tenancy: one system, many clients

Every client (tenant) shares the code but gets partitioned data and per-client config. Two iron rules follow:

  • Every query carries the client ID. Forgetting it returns another tenant's data — a correctness bug at best, a breach at worst.
  • Behavior differences come from per-client config, not per-client code. New client ≈ configuration, not development.

Probably not — uniqueness is per client. (ref + client) is the real key. This is why triage queries that skip the client filter find "impossible" duplicates.

Identity & security — your business domain

This is what makes you a domain expert, not just a Java developer. All of it is public-standards knowledge (NIST, RFCs, FIDO) — the exact standards your platform implements.

Foundations

Authentication vs authorization

AuthN: who are you? (password, OTP, passkey). AuthZ: what may you do? (roles, permissions, subscriptions). They fail differently: 401 = we don't know who you are; 403 = we know exactly who you are, and no.

AuthZ — identity succeeded; entitlement didn't. Look at roles/subscriptions/permissions, not credentials. Classifying the ticket this way instantly halves the search space.

Foundations

How passwords are stored (never stored)

Systems store a salted hash, not the password: hash(salt + password). Login re-computes and compares. Hashes are one-way; the salt (random, per-user) defeats precomputed rainbow tables; slow algorithms (bcrypt/PBKDF2/argon2) make brute force expensive.

It doesn't exist anywhere — only its hash does, and hashing is one-way by design. That's why every flow is reset-based, never recover-based.

MFA

The three factors

Something you know (password/PIN) · something you have (token, phone, passkey device) · something you are (biometric). MFA = at least two different factors. Password + security question = still one factor (two knows).

Weakly at best: email is protected by… often another password, so the "have" is questionable. Regulations for controlled-substance prescribing require a hard token, TOTP app, or biometric as the second factor — which is exactly why your platform exists.

MFA

OTP mechanics: HOTP vs TOTP

Both sides share a secret key. The device computes code = truncate(HMAC(secret, counter)).

  • HOTP (RFC 4226): counter = number of uses. Devices can drift ahead of the server → the server checks a look-ahead window; too far apart needs a resync (enter two consecutive codes).
  • TOTP (RFC 6238): counter = current time ÷ 30s. Drift = clock skew; servers accept ± a step or two.
  • Hardware tokens and authenticator apps are the same math in different packaging.

(1) Counter/clock drift beyond the window → needs resync. (2) Server-side lockout from accumulated failures → the failures and lock state, not the token, block it. The audit trail distinguishes them: wrong code events vs locked events.

MFA

Phone (OOB) and push — convenience with caveats

  • OOB (out-of-band): server sends a code via SMS/voice — possession of the phone number. Weakness: SIM-swap and forwarding attacks; why high-assurance flows restrict it.
  • Push approval: server sends a signed challenge to an enrolled app; user taps approve; app signs back. Stronger than SMS (device-bound), phishable via "approve fatigue".
  • Both are staged transactions: request created → sent → user acts → server completes or expires it. Stuck = find which stage.
MFA

Passkeys / FIDO2 — why they're the endgame

Enrollment creates a key pair on the device: private key never leaves (secured by biometric/PIN); server stores only the public key. Login: server sends a random challenge → device signs it → server verifies with the public key.

  • Phishing-resistant: the signature is bound to the site's origin — a fake site gets a signature that's useless for the real one.
  • Nothing secret is stored server-side → nothing to breach and replay.

An OTP is a bearer secret — whoever holds the 6 digits can use them anywhere. A passkey signature is origin-bound and single-challenge: it only answers this site's random challenge, so relaying it elsewhere proves nothing.

Signing

PKI & digital signatures — how document signing works

document ──hash──► digest ──sign with PRIVATE key──► signature
verify:  recompute digest + check signature with PUBLIC key (from cert)
cert = "this public key belongs to Dr. X" — vouched by a CA chain
  • Sign the digest, not the document — fast, and the doc itself never needs to leave the client.
  • Any change to the document changes the digest → the signature no longer verifies. That's integrity.
  • Private key on the signer's side only → non-repudiation: only they could have signed.

Privacy and efficiency: the platform can bind a signature to the exact document without ever seeing patient data — and the signature still proves integrity, because any change to the prescription would change its digest.

Federation

SSO: SAML, OIDC, and trust

Federation = one system (IdP, identity provider) vouches for users to others (SP, service providers). The user authenticates once; the IdP hands the SP a signed assertion (SAML XML) or signed token (OIDC/JWT). The SP trusts the signature, not the user.

  • SAML: enterprise-era XML assertions via browser redirects. OIDC: JSON/JWT on OAuth2 rails. Same trust shape.
  • Debug SSO by asking: which side minted the assertion, what claims are in it, did the signature/audience/clock checks pass?

The SP doesn't accept the assertion (bad audience/clock skew/missing claim) and silently redirects back to the IdP, which sees a valid session and immediately returns — loop. Read the assertion's claims first, not the login form.

Proofing

Identity proofing — IAL levels (NIST 800-63A)

Authentication proves you own the credential. Proofing proves you are a real, specific person before issuing it.

  • IAL1: self-asserted (you say you're Bob). IAL2: evidence-verified (documents, records, or supervised remote check). IAL3: in-person, strongest.
  • Methods you'll recognize: KBA (knowledge questions from credit-bureau records), document + selfie upload, live webcam review by a trained agent, mailed one-time codes to an address of record.
  • Proofing is a multi-step transaction with its own state machine — resumable, expirable, reviewable. High-value credentials (e-prescribing) require completed proofing before activation.

The signature legally asserts "a proofed, verified person signed this." Mid-proofing, that claim isn't true yet — so the gate stays closed until the process completes. Compliance logic expressed as a state guard.

Defense

Lockouts, fail counters, and unlock flows

Brute-force defense: count consecutive failures per user/credential; past a threshold, lock until a timestamp or until admin action. Success resets the counter.

  • Triage reads: fail count (how close to lock), lock-until (when it clears), and the audit trail (what failed, when, from where).
  • Sibling systems can mirror lock state to each other (a sync daemon) — "locked in one place, fine in the other" means the mirror lagged or failed.
  • Design subtlety: increment must be atomic (see the SQL transactions card) or concurrent failures under-count.

Code craft — reading, debugging, thinking

The skills seniors actually run on. Every one is trainable, and every one has a protocol — no talent required, only reps.

Reading

Why you blank at code (and the fix)

Reading line-by-line tries to hold the whole file in working memory — it overflows and the mind whites out. Seniors read in chunks ("that's a factory dispatch", "that's validate-then-persist") and always with a question. Chunks come from reps; questions come from habit.

  • Before opening any file, write ONE question it must answer.
  • Find the answer, write file:line → answer, close the file.
  • Blanking anyway? Shrink scope: one method → one statement, said out loud.

It has no stopping condition — you'll read everything and retain nothing. "Where does the status get set to ACTIVE?" is answerable, checkable, and done in minutes. Good questions have findable answers.

Reading

The two-pass method

  1. Pass 1 — happy path only. Skip every if (error) branch. What happens when everything works?
  2. Pass 2 — error branches only. Half of L3 knowledge lives here: what can fail, what each failure returns, what gets logged.

Then sketch the call chain on paper: Controller.method → Service.method → Helper.method → table. The sketch is the deliverable — if you can't draw it, you haven't read it yet.

Debugging

The triage ladder (same shape as the DSA protocol)

  1. Restate: what exactly failed, for whom, since when?
  2. Tiny example: one user, one timestamp, one request
  3. Timeline: audit/event rows around that moment
  4. Code: open the one method that writes those events — with one question
  5. State: does the DB row agree with what the code assumes?
  6. Blast radius: count — one user or thousands?

Restate → small example → trace → pattern. Debugging prod and solving LeetCode are the same skill in different clothes. Every rep trains both.

Thinking

The senior checklist (run it on every change)

  • Nulls: which of my inputs can be null/empty/absent, and what happens then?
  • Concurrency: two requests hit this at once — what breaks?
  • Duplicates: this message/request arrives twice — is the result the same?
  • Time: timezone? clock skew? what if the job runs at midnight/DST?
  • Blast radius: if I'm wrong, who is affected and how do we know?
  • Rollback: how do we undo this — code revert, config flip, data fix?
  • Observability: when it misbehaves at 2am, what will the logs show?

Duplicates (at-least-once delivery guarantees them) and concurrency (fail counters, status transitions, sync daemons all race). Nulls are third — DB columns are nullable in ways code forgets.

Thinking

Reading other people's PRs (fastest senior-maker there is)

  1. Read the ticket first — what was supposed to change?
  2. Find the one file that carries the real change; the rest is ripple.
  3. Ask the checklist questions (nulls, concurrency, duplicates) of the new code.
  4. Write one genuine question as a comment. Asking beats silence — and beats fake approval.

You learn more from 20 minutes in a teammate's PR than from an hour of tutorials: it's real code, real constraints, and the review discussion shows you how seniors think.

Habit

The learning loop that makes it stick

  • Type, never paste. Pasting is reading with extra steps.
  • Recall, never reread. 24h after learning X, write it from a blank page. Failing the recall IS the study — that's when memory forms.
  • Connect within 48h. Every concept gets found in real code at work within two days, or it evaporates.
  • Missed days don't exist. Never catch up, never restart — just do today.

Deep dive — intermediate & advanced

The layer below the framework: how the machinery actually works, and the failure modes seniors recognize on sight.

Spring internals

Proxies: how @Transactional really works

Spring doesn't change your class — it wraps the bean in a proxy that intercepts calls: open transaction → call your method → commit/rollback. Everything annotation-driven (@Transactional, @Async, security annotations, caching) rides this mechanism.

caller ──► [proxy: begin txn] ──► yourService.bindCredential() ──► [proxy: commit]

yourService.bindCredential() ──► this.helper()   // stays INSIDE the object —
                                                  // the proxy never sees it
  • Self-invocation (this.method()) bypasses the proxy → annotations on that method are ignored.
  • Proxies wrap public methods; @Transactional on private/final methods silently does nothing.

Same proxy rule as @Transactional: the async dispatch lives in the proxy, and internal calls never cross it. Move the method to another bean or inject the bean into itself.

Spring internals

Transaction propagation & rollback rules

  • REQUIRED (default): join the caller's transaction if one exists, else start one. One failure rolls back everything joined.
  • REQUIRES_NEW: suspend the current transaction, run an independent one that commits on its own — how audit rows survive a rolled-back business operation.
  • Rollback happens on unchecked exceptions by default. A checked exception thrown through @Transactional commits unless you say rollbackFor.

Write the audit row in a REQUIRES_NEW method (on a different bean — self-invocation!). The inner transaction commits independently; the outer one can still roll back the business writes.

Plumbing

Connection pools — the resource everyone forgets

Opening a DB connection is expensive, so apps keep a fixed pool (10–50). Every query borrows one and must return it. The two classic failures:

  • Leak: a code path exits without returning the connection (missing finally/try-with-resources). The pool drains over hours → every request suddenly hangs on "waiting for connection." Restart "fixes" it — until tomorrow. This is exactly why legacy code religiously calls its finalize-datasource in finally.
  • Exhaustion under load: slow queries hold connections longer → pool empties → timeouts cascade. The DB looks "down" but the bottleneck is the pool.

Connection leak. Find the code path that borrows without a guaranteed return — usually an early return or exception path skipping cleanup. Pool metrics (active vs idle climbing to max) confirm it.

Plumbing

Resilience: timeouts, retries, circuit breakers

Every remote call needs three decisions (your services use Resilience4j for these):

  • Timeout: how long to wait before giving up. No timeout = one slow dependency freezes your thread pool.
  • Retry: try again on transient failure — but ONLY if the operation is idempotent, or retries duplicate it.
  • Circuit breaker: after N failures, stop calling for a while and fail fast — protects both sides from a death spiral.

The first attempt may have succeeded (response lost in transit). Retry = second credential. Either make creation idempotent (client-supplied unique ref, upsert semantics) or don't retry non-idempotent writes.

Concurrency

The real toolbox

  • synchronized — one thread at a time; simple, coarse. Legacy singletons live on this.
  • volatile — visibility only: threads see the latest write. NOT atomicity: volatile int x; x++ still races (read-add-write).
  • AtomicInteger.incrementAndGet() — atomic counters without locks.
  • ConcurrentHashMap — safe concurrent map; computeIfAbsent is atomic per key. The cache building block.
  • Tomcat's worker pool IS your concurrency ceiling: ~200 threads by default. Block them all (slow DB, missing timeout) and the service is "down" while the process is fine.

(1) AtomicInteger, (2) synchronized block, (3) best for your platform: push it into the DB — UPDATE … SET cnt = cnt + 1 serializes on the row lock and survives multiple service instances, which in-JVM fixes don't.

JVM

Memory & GC — enough to read an incident

  • Heap = young generation (new objects, collected fast + often) + old generation (survivors). Full GC pauses = old gen pressure.
  • OutOfMemoryError: Java heap space — something holds references forever: unbounded caches/maps, giant result sets (SELECT * from a million-row table into a List), accumulating listeners.
  • Symptom curve of a leak: sawtooth heap graph whose valleys climb day over day, then GC storms (CPU pinned, requests slow) before the OOM.
  • First questions: what grew? (heap dump / histogram: which class has millions of instances) and what changed? (deploy, traffic, data volume).

Leak-shaped: load-only pressure recovers when traffic drops; climbing valleys after GC mean retained objects accumulate. Confirm with two heap histograms an hour apart — the leaking class's instance count only goes up.

Database

Blocking vs deadlocks

  • Blocking: A holds a lock, B waits. Resolves when A commits. Long transactions + hot rows = timeouts that look random.
  • Deadlock: A waits for B's lock while B waits for A's — circular. The DB picks a victim, kills it, the other proceeds. The victim's error is retryable by design.
  • Prevention: touch tables in a consistent order, keep transactions short, never hold a transaction across a remote call.

The row locks are held for the whole remote call — seconds, or forever on a hang. Every other writer of those rows queues behind it. One slow dependency becomes a database-wide pileup.

Database

Isolation levels — what readers see mid-write

  • READ COMMITTED (SQL Server default): readers see only committed data, but two reads in one transaction can differ (non-repeatable).
  • READ UNCOMMITTED / NOLOCK: reads uncommitted, possibly-rolled-back data. Fine for rough triage counts; never for logic.
  • Snapshot/RCSI: readers see a consistent point-in-time photo; writers don't block readers.

Under NOLOCK: an uncommitted write that later rolled back — data that never officially existed. Or simply a concurrent delete between reads under READ COMMITTED. Either way: re-verify before building an RCA on one read.

Database

Index design — composite order matters

INDEX (CLIENT_ID, STATUS, CREATE_TIME)

WHERE CLIENT_ID = ? AND STATUS = ?          ✓ seek (leftmost prefix)
WHERE CLIENT_ID = ?                          ✓ seek
WHERE STATUS = ?                             ✗ scan — skipped the first column
WHERE CLIENT_ID = ? ORDER BY CREATE_TIME     ✓ seek + no sort needed
  • A composite index is sorted by column 1, then 2, then 3 — like a phone book (last name, first name). You can't look up by first name alone.
  • Covering: if the index contains every column the query needs, the table itself is never touched — the fastest query is one that never leaves the index.
  • Equality columns first, range/sort columns last.

Indexes on tenant tables lead with CLIENT_ID — providing it gives every query the leftmost prefix and a seek. Skipping it isn't just a data-leak risk; it's a full scan of everyone's rows.

APIs

Evolving APIs without breaking clients

  • Safe (additive): new optional field, new endpoint, new enum value you emit.
  • Breaking: renaming/removing a field, changing a type, making optional required, new required input.
  • Jackson default: unknown fields in a response are ignored by clients — which is what makes additive change safe. Strict deserializers break on them; know which your consumers use.
  • When breaking is unavoidable: version the path (/v1/ → /v2/) and run both — which is precisely why platforms accumulate parallel API versions for years.

You can't see who still reads it. Integrations parse responses on their schedule, not yours — the field is a contract until every consumer confirms otherwise (or telemetry proves zero readers).

System design — the platform below the code

Networking, high availability, HSMs, and the infrastructure fork. Grounded in the platform's real architecture docs, kept at pattern level. This is what turns "the service returned 502" from a mystery into a checklist.

Networking

Life of a request — the network half

browser ─► DNS ─► edge (load balancer / WAF)
                    │  TLS usually TERMINATES here
                    │  adds X-Forwarded-For: <real client IP>
                    │  adds X-Client-Cert:   <user's cert, if presented>
                    ▼
              reverse proxy ─► app host (Tomcat) ─► DB
  • Once TLS terminates at the edge, the app sees a connection from the load balancer — that's why the client's real IP must ride in X-Forwarded-For, and a presented client certificate rides in a header like X-Client-Cert.
  • Consequence: the app trusts the edge for those headers. If a request could reach the app without passing the edge, those headers could be forged — which is why the network is designed so it can't.

No — that's the load balancer's address, because it terminates the client connection and opens its own to the app. The real client IP is in X-Forwarded-For. Logging that header (and knowing it) is basic triage hygiene.

Networking

TLS and mutual TLS (mTLS)

  • TLS: the server proves its identity with a certificate; traffic is encrypted. The browser padlock.
  • mTLS: BOTH sides present certificates — the client proves its identity too. This is how API partners and internal system-to-system calls (portal ↔ MFA engine) authenticate.
  • Certificates expire. A forgotten renewal is the single most classic "everything broke at midnight, nothing was deployed" outage — and cert serial/case mismatches are a recurring integration bug class.

(1) An expired or rotated certificate on either side; (2) the trust configuration — their new cert isn't registered/trusted (or a mismatch in how the cert is recorded). Handshake failures are almost never application code.

Networking

Egress: outbound is locked down too

In a hardened environment, servers can't call the internet freely. All outbound traffic routes through a proxy that is deny-by-default: only whitelisted destination FQDNs are allowed, and the proxy masks the caller's internal IP.

Egress rules — the destination FQDN isn't on the outbound proxy's whitelist yet. Not DNS, not code. The fix is a firewall/proxy change request, and knowing that saves a day of code-side debugging.

Pattern: "works locally, fails on the server" for outbound calls = egress. "Works on the server, fails for users" for inbound = edge/WAF. Direction tells you which wall to inspect.
Availability

High availability: the two-leg model

            ┌─► leg A: proxy ─ app hosts ─┐
edge/LB ────┤                             ├─► SQL Always-On
            └─► leg B: proxy ─ app hosts ─┘    (primary + sync replica,
                (physically & logically         automatic failover)
                 separate beyond the proxy)
  • Production runs two independent legs; each can carry the load alone. Databases use Always-On availability groups — a synchronized replica takes over on failure.
  • The payoff beyond surviving failures: drain one leg, deploy to it, shift traffic back — anytime deployments and maintenance with zero downtime.
  • Design question to ask of any component: "what happens when exactly one of these dies?" If the answer is "everything stops," you've found a single point of failure.

One leg is sick — bad deploy, dead host, or config drift on that leg only. Check which leg the failing sessions land on, then drain it. "Exactly half affected" is a topology clue, not a coincidence.

Crypto infrastructure

HSMs: where the keys really live

An HSM (hardware security module) is a tamper-resistant device that holds cryptographic keys which never leave it — you send it data, it returns signatures/validations. The platform's OTP validation and prescription-signing keys live in HSM-backed infrastructure; cloud Key Vaults are the managed equivalent for newer components.

  • Why: compliance (the signing key for controlled-substance prescriptions must be provably uncopyable) and breach containment (a stolen server image contains no keys).
  • Cost: HSMs are physical — capacity, redundancy, and geo-resilience require hardware planning, which is why "just run a second copy" is a project, not a checkbox.

A key in config can be read, copied, and exfiltrated — and you could never prove it wasn't. Non-repudiation ("only this system could have signed") requires the key to be physically incapable of leaving the module. Regulation demands that property.

Federation

The SSO hub — and strangler at the network layer

A federation server (PingFederate) sits at the center: users authenticate once, and it mints signed assertions for every connected application (the Identity tab's IdP/SP card, deployed). Two system-design lessons from its operation:

  • Each partner/application is a connection with its own certificates, claims mapping, and lifecycle — federation outages are usually one connection's config, not the hub.
  • During platform migrations, the load balancer can route per connection — old federation stack for most, new stack for migrated ones. The strangler pattern, implemented in network routing: migrate one integration at a time, big-bang never.
Operations

Observability: is it down, slow, or lying?

  • Health endpoints (/actuator/health) answer "is the process up and can it reach its dependencies?" — the first curl of every incident.
  • Centralized logs (Splunk / log analytics) answer "what happened?" — searchable across hosts, because you'll never RDP to eight machines fast enough.
  • Metrics answer "how is it trending?" — error rates, latencies, pool usage. Slow is a different disease than down.
  • Correlate across services with shared IDs (trace/error refs in responses map to log entries) — that reference in an error response is the thread to pull.

The process is up but a specific path is broken: one endpoint erroring, one dependency degraded (health checks it shallowly), one leg bad behind the LB, or auth failing before requests reach the app. Green health narrows the search — it doesn't end it.

Operations

CI/CD + Infrastructure as Code

  • Pipelines: build-and-publish (compile, test, version the artifact) → deploy (push that exact artifact per environment). The same artifact climbs DEV → QA → UAT → PROD — promotion, never rebuilding per environment.
  • Infrastructure as Code (Terraform): the environment itself is versioned code — plan shows the diff on a PR, apply executes on merge. Clicking around a cloud console creates drift nobody can reproduce.
  • Why parity matters: every "works in UAT, fails in PROD" is an environment diff — config value, cert, firewall rule, data shape. IaC shrinks that list; it never empties it.

Rebuilds can differ (dependency drift, build-time config, timestamps). Promoting the identical binary means what you tested IS what ships — the only variables left are environment config and data.

Scale & recovery

Scaling and disaster recovery

  • Horizontal scaling (more instances) requires statelessness — any instance must serve any request. That's the deep reason services keep no per-user state in memory (the Java thread-safety card, at system scale).
  • The database is the shared bottleneck: 10 instances × 30-connection pools = 300 connections hitting one primary. Scaling apps without watching the DB just moves the queue.
  • DR vocabulary: RTO = how long you may be down; RPO = how much data you may lose. Geo-redundancy (customer-driven requirement) means surviving the loss of an entire site — active/active or a warm standby region.
  • A DR plan that's never been exercised is a document, not a capability.

If the DB (or a downstream dependency) is the bottleneck, more instances add more connections and more retry pressure on the thing that's drowning. Scale the constraint, not the thing in front of it.
